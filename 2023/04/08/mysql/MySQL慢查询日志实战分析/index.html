MySQL 慢查询日志实战分析

从 0 到 1 实战排查慢 SQL

一、什么是慢查询日志？

慢查询日志（Slow Query Log）是 MySQL 提供的一套 SQL 性能诊断机制，用于记录：

  • 执行时间超过阈值的 SQL
  • 未走索引的 SQL
  • 锁等待时间过长的 SQL

一句话：慢查询日志 = 性能问题定位神器

二、开启慢查询日志（生产必备）

1. 查看当前配置

重点关注参数：

参数 含义
slow_query_log 是否开启慢查询
slow_query_log_file 慢查询日志文件路径
long_query_time 超过多少秒算慢
log_queries_not_using_indexes 是否记录未走索引 SQL

2. 开启慢查询日志

生产推荐：long_query_time = 0.5

互联网高并发系统，0.5s 已经算慢。

核心指标解析：

字段 含义
Query_time SQL 执行耗时
Lock_time 锁等待时间
Rows_examined 扫描行数
Rows_sent 返回行数

关键优化指标：Rows_examined / Rows_sent 比例越大 → 扫描越多 → 性能越差

理想值：< 100

四、慢查询日志分析工具

方案一：mysqldumpslow（自带）

参数 作用
-s t 按时间排序
-t 10 取前 10 条

方案二：pt-query-digest（企业级）

Percona 官方神器（强烈推荐）

输出分析报告，包含：

  • TOP 慢 SQL
  • 平均耗时
  • 调用频率
  • 95 分位耗时
  • 扫描行数统计

Step 2：EXPLAIN 分析执行计划

重点看：

字段 优化目标
type ref / range / const
key 是否命中索引
rows 扫描行数
Extra 是否 Using index

Step 3：判断慢因类型

类型 典型特征
索引缺失 type=ALL
索引失效 使用函数 / 隐式转换
深分页 LIMIT 100000,10
大表 Join Nested Loop
锁冲突 Lock_time 高

Step 4：制定优化方案

问题 解决方案
无索引 补索引
索引失效 改 SQL
深分页 子查询 + 覆盖索引
Join 慢 先过滤再 join
锁冲突 拆事务 / 降隔离级别
